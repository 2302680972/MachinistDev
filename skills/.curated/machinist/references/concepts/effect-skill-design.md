# 特效和技能设计原则

这篇写的是可落地的做法：在现有零件、现有脚本接口的范围内，把“技能效果”和“表现特效”做成可控、可复现、可调参的结构，而不是幻想一套独立的技能系统。

## 本篇范围

- 什么是技能、什么是特效，以及两者的边界
- 技能的组织方式与常见触发来源
- 冷却、次数、节流这类高频约束
- 读表做参数化时的取舍

## 不包含

- UI 动画细节与实现机制
- 图片、动图、视频类表现的具体落地

## 先把概念分开

### 技能是什么

技能是脚本侧的一段“可重复触发的效果逻辑”。它至少需要三件事：

- 触发来源：来自事件、弹幕消息、礼物、碰撞、阶段节点等
- 结算逻辑：真正改变局内状态的部分，例如造成伤害、生成投射物、改单位属性、改排行榜分数
- 约束：冷却、次数、持续时间、同屏上限、按观众或按单位的配额

### 特效是什么

特效是“表现”，本质是给观众看的反馈。它可以是音效、世界中的爆炸表现、粒子零件效果、UI 文本特效、环境开关等。

本项目里常见的可用入口包括：

- 音效与爆炸表现：`G.audio.playEffect`、`G.audio.explode`
- 环境类开关：`G.effect.switchVolumeFog`
- UI 文本特效：`UILabel.labelEffect`

不同工程拥有的零件与资源不同，特效能做到什么，以工程里实际能调用到的接口与能用的零件为准。

## 设计原则

### 先可控，再华丽

- 先做“能稳定触发、能稳定结算”的最小技能，再加表现
- 表现缺失时要能退化，不影响玩法结算
- 任何高频技能都要先考虑同屏上限与节流，再考虑细节

### 逻辑与表现分层

推荐把技能拆成两层：

- 逻辑层：只做结算与状态修改，不关心特效资源名与具体播放方式
- 表现层：只根据“技能已触发”“命中发生”“阶段切换”等信号播放特效

这样做的收益是：改表现不会误伤玩法，调参也更直接。

### 参数化只做数值与开关

读表很适合放这些内容：

- 冷却、持续时间、触发概率、伤害系数、数量上限、范围半径
- 不同礼物档位对应的强度倍率
- 表现强度的倍率，例如爆炸缩放、音量、UI 颜色

不建议把这些塞进表里：

- 把整套技能流程拆成一堆字符串再动态拼接执行
- 把“目标选择规则”写成难以调试的表达式

## 触发来源与边界

### 弹幕与礼物触发

弹幕消息通过 `onDanmu` 事件进入脚本。常见做法是：

- 先把消息解析成"技能请求"，只保留需要的字段
- 按观众维度做冷却与节流，避免单个观众刷屏把逻辑打爆
- 礼物可做强触发，评论与点赞做弱触发，避免免费输入直接决定胜负

### 单位与投射物触发

单位、投射物这种高频实体，常见触发来自碰撞、检测、寿命到期。原则是：

- 一次命中只结算一次，避免同一帧多路事件重复算账
- 命中表现尽量走“轻量特效”，重型表现必须有数量上限
- 投射物优先复用对象池，避免频繁创建销毁

## 冷却与节流：用时间戳更稳

冷却有两种常见表示法：

1) 剩余冷却
- 每次触发写入一个剩余值
- 每帧用 `G.time.deltaTime` 递减
- 适合本来就必须每帧更新的对象

2) 下次可用时间
- 触发时记录 `nextReadyTime = G.time.time + cooldown`
- 判断时用 `G.time.time` 直接比较
- 更适合弹幕消息这种“事件驱动”的技能，不需要为了冷却专门跑每帧递减

无论哪种方式，都建议把 key 设计清楚，常见组合是：

- 按观众：`uid + skillKey`
- 按单位：`unitId + skillKey`
- 按全局：`skillKey`

## 特效落地建议

### 表现优先级

在性能与一致性上，推荐按下面顺序组合表现：

1. UI 提示与文本特效，成本低，反馈快
2. 音效，成本低，强化反馈
3. 少量世界特效，严格受上限控制
4. 大范围环境效果，只在阶段节点或稀有技能使用

### 同屏上限与合并

高频技能的特效要有“合并策略”，典型做法：

- 同一位置短时间内只播一次大特效，其余用轻量提示
- 同一观众的连发礼物只累计数值，表现按批次合并播放

## 调试习惯

- 先把触发条件与关键参数用日志打通，再加特效
- 调试开关建议集中管理，避免把大量输出散落在各个分支
