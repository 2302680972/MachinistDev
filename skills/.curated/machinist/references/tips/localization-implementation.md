# 游戏本地化（工程侧做法）

这份说明面向“游戏工程内容”（脚本/表格/UI 逻辑），不讨论原版客户端本体里那些只对内置 UI 生效的行为。

## 本篇范围

- 语言表结构与脚本读取方式
- UI 文案刷新策略

## 不包含

- 通用读表规则与表格组织（见“读表管理属性”）
- UI 组件引用与运行时规则（见“UI 组件管理指引”）

## 1) 语言表（翻译表）是什么

工程里只有一张语言表，入口由工程根目录的 `setting.json` 指定。

本篇后续提到的“语言表”，都指 `setting.json` 指向的那一张表，不要做目录扫描，不要假设存在多份语言表文件。

语言表用 `name` 做 key，不同列存放不同语言的文本。

表头明确必须这么写,这是规范!!!!绝对禁止又假设不知道或者瞎编不存在的东西!!!:

- `id`: ID(float,必须要有)
- `name`：key（string,稳定标识）
- `cn`：中文(string)
- `en`：英文(string)

如果要做更多语言，一般做法是新增列，并用语言名对应（例如 `Japanese` / `Korean` 之类），脚本侧不需要额外改写。
语言的标识符是固定的,必须用cn和en而不是其他的瞎编的内容
前三行必须仍保持按`table-based-attribute.md`的要求写!!!!

静态 Canvas 里的文本也可以写成 `@条目名`(但不推荐)，由 Canvas 创建时按语言表 `name` 做一次静态翻译；这条链路不走运行时翻译函数，预览也不会去解析语言表，所以预览里会保留原始 `@条目名`。

## 2) 脚本里怎么取本地化文本

工程脚本侧的主入口是“按 key 翻译”：

- 当前语言由平台基座决定，工程脚本只能读取并按当前语言翻译，不能提供“自选语言”逻辑，也不能假设可以在工程侧切换语言。
- `G.string.translate(key)`：把 `key` 翻译成当前语言的文本。`key` 直接填语言表 `name`，不要带 `@`；条目不存在会直接报错，所以语言表要补齐。
- `G.TranslateEx(key, p1..p5)`：先按 `key` 翻译，再把结果里的 `%1`~`%5` 替换成参数。参数如果是以 `@` 开头的字符串，会先按静态翻译规则再翻译一次。
- `G.map.translateEX(key, args)`：旧高级翻译接口，参数放在 `List` 里，行为和 `TranslateEx` 一致但不推荐再用。

## 3) UI/提示如何接入

工程侧通常是"在脚本里给 UI 文本赋值"，所以推荐写一个"刷新文案"的函数：

- 打开面板/进入地图/关键状态变化时，统一调用一次刷新，把所有文案重新 `translate` 后再赋值。
- 不要依赖“语言切换时 UI 自动更新”这类行为；工程侧更可控的做法是主动刷新。

## 5) 内容侧建议

- `name`（key）尽量用稳定的标识（例如 `ui.loadLevel`），别把整句自然语言当 key（易改、易重复、难管理）。
- 先把“需要本地化的文本”全部收敛到语言表，再让脚本只引用 key；这样改文案/补语言不会牵扯到逻辑。
