# 性能优化考量

## 本篇范围

- 性能瓶颈的常见来源
- 实用的减压策略与反模式

## 不包含

- 投射物与碰撞的具体规则（见"projectile-collision-detector"）
- UI 动画的具体实现（见"canvas-animation-mechanism"）

## 主要瓶颈

1. 频繁创建销毁对象
2. 每帧大量计算
3. 过多碰撞检测
4. UI 频繁更新

## 对象池

预创建对象池，需要时获取激活对象，失效后回收而非销毁。维护活跃对象列表便于遍历更新。

降低更新频率（跳帧）：不是所有逻辑都需要每帧更新，可以设置更新间隔（如每 5 帧更新一次）。

## 空间划分

用网格划分场景，每个网格存储该区域内的对象。

碰撞检测时只检查：
1. 对象所在网格的对象
2. 周围 8 个相邻网格的对象

避免全局范围检测。

## 距离剔除

先粗略检查距离（避免开方运算）：比较 (dx² + dy²) 和 (range²)，超出范围的对象直接跳过处理。

## 保存昂贵的计算结果

当某些计算确实昂贵且会被重复用到时，可以把结果保存到 `dict` 里，后续先查再算。

约束：

- 不要先假设“必须做缓存”，先确认瓶颈点再动手。
- key 必须稳定且可解释，避免命中不稳定导致结果混乱。
- 必须有清理策略，避免数据无限增长。

## 数量控制

- 弹幕玩家：设置上限（50-100），超出时移除最早或不活跃的
- 子弹：限制同屏数量，超出时回收最早的
- UI：避免过多动态组件，复用组件

## 监控

维护帧计数器，每秒输出一次 FPS。通过 FPS 判断性能是否正常。

## 反模式

- 每帧创建新对象
- 嵌套循环遍历所有对象对
- 全局范围检测所有碰撞
- 不必要的 UI 属性修改
